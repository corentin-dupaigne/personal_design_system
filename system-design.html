Design System

A unified token, component, content, and governance system for web, app, software, and slide contexts. All monospace, warm parchment, minimal — across light and dark.

IBM Plex Mono Warm parchment Light + dark WCAG AA Multi-platform
00 — Foundations

Principles

Six rules. Every token, component, and pattern in this system is downstream of these. When something feels off, check whether it violates one of them — usually it does.

01
Meaning over decoration
Color, weight, and motion exist to encode meaning, never to ornament. A red border means failure; an accent color means active. If it doesn't say something, it shouldn't be there.
02
Monospace as identity
One typeface. Hierarchy comes from weight, size, and spacing — not from font variety. The all-mono stack is non-negotiable; it's what makes the system recognizable across web, slides, and print.
03
Warmth over contrast
No pure black, no pure white. Every surface holds warmth. Elevation is expressed through tonal shifts in the parchment, not drop shadows or hard contrast.
04
Functional motion
Animation reveals state — focus, transition, dismissal. Never decorative bounce, never delay over 250ms. One easing curve throughout. Reduced-motion preferences are honored at the token level.
05
One way to do a thing
If two components could be used for the same job, only one is correct. The component catalogue is small on purpose. Adding variants requires governance — see Governance.
06
Accessible by default
All token pairings meet WCAG AA contrast. Focus is always visible. Motion is reducible. Touch targets are at least 44px. Components ship accessible — there is no "a11y pass" later.

Theming

The system ships in light and dark. Components consume tokens, not hex — switching themes is a single attribute change. Use the toggle at the top of this page to see every section re-skin live.

Activation
// 1. Default — follows OS preference automatically
<html> <!-- prefers-color-scheme handles it -->

// 2. Force light or dark
<html data-theme="light">
<html data-theme="dark">

// 3. JS toggle
document.documentElement.dataset.theme =
  document.documentElement.dataset.theme === 'dark' ? 'light' : 'dark';
Token contract

Every token has a value in both themes. Components must reference tokens (var(--fg)) — never raw hex. This is the single rule that makes theming reliable.

--bg → --fgPage → ink. Inverts across themes; always > 11:1 contrast.
--bg-subtle … --bg-overlayElevation steps. Each darkens (light theme) or lightens (dark theme) by ~5% lightness.
--accentLight: #b8882a · Dark: #d8a85c. Brightened in dark to maintain perceived saturation.
Semantic (success/warning/danger/info)Hue preserved across themes; lightness adjusted to maintain ≥ 4.5:1 against the surface they sit on.
--shadow-*Light: warm-foreground tinted at low alpha. Dark: pure black at higher alpha — parchment shadows disappear on dark surfaces.
Authoring rules
Do
color: var(--fg);
References a token. Re-themes automatically and never drifts.
Don't
color: #2c1f0e;
Hex pinned to one theme. Looks fine in light, invisible in dark.
Do
Test every component in both themes during review. Dark mode is not optional.
Don't
Use a different layout, density, or radius in dark mode. Theming changes color only.
02 — Foundations

Accessibility

WCAG 2.2 AA is the floor, not a goal. Every token pair, focus state, motion preference, and touch target in this system has been chosen with this in mind. Components ship accessible — there is no "a11y pass" later.

Contrast — light theme

All token pairs used for body text or actionable elements meet AA (≥ 4.5:1 for body, ≥ 3:1 for large / non-text).

--fg on --bg

--muted on --bg
4.62:1AA
--accent on --bg
4.51:1AA

5.06:1AA

Focus state

Every interactive element shows a 2px --accent outline with 2px offset on keyboard focus. Mouse focus is suppressed. Focus is never removed without a replacement.

Focused (keyboard)
Unfocused
Keyboard navigation
Tab / Shift+TabMove forward / backward through interactive elements in DOM order.
Enter / SpaceActivate buttons, submit forms, toggle expanded state on disclosures.
EscClose modals, dropdowns, popovers, toasts. Always dismisses overlays.
↑ / ↓Move within lists: dropdown items, autocomplete options, radio groups.
← / →Move within tabs and segmented controls. Update selection on move.
Home / EndJump to first / last item in a list, tab strip, or pagination control.
⌘+KOpen command palette (when present in the app shell).
Touch targets

Minimum 44 × 44px hit area for any interactive element on touch surfaces — even when the visible component is smaller. Use invisible padding to expand the target without changing the visual size. The token --touch-min is the source of truth.

Reduced motion

When the user has set prefers-reduced-motion: reduce, all transitions and animations collapse to 0.001ms. This is wired globally — components don't need to handle it individually. Skeletons stop shimmering, toasts appear without sliding, smooth-scroll becomes instant.

Screen reader guidance
Decorative iconsaria-hidden="true". They mustn't reach the screen reader.
Meaningful iconsrole="img" + aria-label="…" with a literal description, not a metaphor.
Buttons that are icon-onlyAlways have aria-label. The visible glyph is not announced.
Live regions (toasts, alerts)role="status" for non-urgent, role="alert" for incidents.
Form inputsAlways paired with a <label>; never rely on placeholder as label.
HeadingsOne h1 per page. Don't skip levels — heading order is the screen reader's table of contents.
Color aloneNever the sole signal. Pair with text, icon, or position. A red dot is not enough.
Forms — error association

Inputs in error state set aria-invalid="true" and link to their hint via aria-describedby. The hint text shifts to --danger and stays in the DOM after the error clears, so screen readers don't lose track of the message.

<label for="threshold">Threshold (ms)</label>
<input id="threshold" aria-invalid="true" aria-describedby="threshold-hint">
<div id="threshold-hint" class="input-hint">Must be a number between 0 and 60000.</div>
Skip links

Every page with a persistent sidebar includes a "Skip to main content" link as the first focusable element. It is visually hidden until focused. This is non-negotiable for keyboard-only users on data-dense layouts.

When to use which
Alert
Persistent state the user needs to acknowledge or that affects their next decision (open incident, error budget low, deployment in flight).
Toast

04 — Identity

Voice & content

A design system without a voice produces inconsistent products. The visual system is monospace and minimal; the writing should match. Direct, technical, and respectful of the reader's time.

Voice — four traits
A
Direct
No hedging. "This will delete the table" instead of "Please note that this action may delete the table." If the action is destructive, say so plainly.
B
Technical, not jargon
Use the precise term when it exists ("p99 latency," "error budget"). Skip marketing words ("seamless," "powerful," "cutting-edge"). The reader knows what good software looks like.
C
Calm in incident
When something fails, the copy stays neutral. "Could not save" — not "Oops!" or "Something went wrong." Panic is the system's job to suppress, not to spread.
D
Respectful of time
Every word earns its place. If a sentence can be cut without losing meaning, cut it. Help text is for facts the user needs — never for friendly chatter.
Capitalization & punctuation
Sentence caseButtons, labels, titles, menu items. Run deploy — not Run Deploy.
UPPERCASESection labels (8pt+, with letter-spacing). Brand wordmarks. Status badges.
No trailing punctuationButtons, labels, headings, bullet lists. Hint text and full sentences in body copy keep periods.
Em dashesUse the typographic em dash (—), not double hyphens. Surround with spaces.
NumbersNumerals always for measurements (2 ms, 99.97%, 14 services). Spell out "one" only in narrative prose.
Time24-hour clock with timezone abbreviation: 03:00 UTC. Never 3am.
DatesISO format in tables and metadata: 2026-04-25. "Apr 25, 2026" only in prose.
Microcopy — buttons
Do
Verb + object: Run deploy, Save runbook, Roll back. The button says what happens when it's pressed.
Don't
OK, Submit, Done. These are placeholders, not actions. They tell the user nothing about consequence.
Do
Confirm the destructive action by name: Delete service, not Yes, I'm sure.
Don't
Use exclamation marks. Ever. Not in buttons, not in toasts, not in alerts. The system isn't excited.
Microcopy — error messages

An error message has three jobs: name what failed, explain why (briefly), tell the user what to do next. In that order.

Do
Could not save runbook — session expired. Sign in again to retry.
Don't
Error 401. The user doesn't read HTTP. Translate.
Do
Threshold must be a number between 0 and 60000. States the constraint.
Don't
Invalid input. Doesn't say which input or what's wrong.
Terminology dictionary

Pick one term, use it everywhere. Inconsistency is the enemy of trust.

serviceAlways "service," never "app," "microservice," or "API" when referring to a deployable unit.
deployVerb and noun. Not "release," "rollout," or "push."
incidentP0/P1/P2 events. Not "outage," "issue," or "problem" in product copy.
silenceThe act of suppressing alerts. Not "mute" or "snooze."
on-callHyphenated, lowercase. Not "oncall" or "On Call."
runbookOne word. Not "run book" or "playbook."
postmortemOne word, no hyphen. Not "post-mortem."
SLO / SLA / SLIAlways uppercase, no periods. Define on first use in user-facing copy.
Internationalization

Strings live in a flat key-value catalogue, never inline. Reserve 30% extra width for German, 50% for Russian. Avoid concatenating sentence fragments — translators need full sentences for grammar agreement. Date and number formatting always go through Intl.* — never hand-rolled.

05 — Operations

Governance

A design system is a contract between teams. These rules describe how the contract is changed, who can change it, and how consumers are notified.

Versioning

Semantic versioning. The token catalogue is the public contract — breaking it bumps the major version.

MAJOR (x.0.0)Token renamed or removed; component public API changed; any visible regression. Migration guide required.
MINOR (1.x.0)New token, new component, new variant. Backwards-compatible visual refinement. No migration needed.
PATCH (1.0.x)Bug fixes, accessibility fixes, doc-only changes. Always safe to upgrade.
Contribution flow
  1. Open an RFC issue with the proposed change, the problem it solves, and three candidate designs.
  2. Discuss in the weekly system review. Decisions are recorded in the issue, not in chat.
  3. If approved, implement on a feature branch. Update tokens, components, docs, and the changelog in one PR.
  4. Run the visual diff suite. Any unexpected diff is a blocker until acknowledged or fixed.
  5. Maintainer review focuses on: naming, accessibility, theming parity, doc completeness.
  6. On merge, a release is cut automatically. Consumers are notified via the release feed.
Deprecation policy
MarkingAdd a @deprecated annotation on the token / component, plus a console warning in dev builds.
Soak windowMinimum two minor versions or 90 days, whichever is longer, before removal.
Migration pathEvery deprecation must name its replacement and link to a codemod or migration recipe.

03 — Slides

Content rules

Rules for what goes on a slide and how it is written. These are as important as visual rules — a well-designed slide with a bad title is still a bad slide.

The insight title rule
NO
Topic title: "Latency analysis" — describes what the slide is about, not what it means.
YES
Insight title: "p99 latency has tripled since the June 10 deploy" — states the finding directly.

The title is the message. A reader who only reads the titles of every slide should understand the full argument of the deck. This is the Pyramid Principle applied to slides.

Structural rules
RULE
One governing thought per slide. If a second point is needed, create a second slide.
RULE
Every number on a slide must have a source. Source goes in the footer left, at 7pt, prefixed with "Source:".
RULE
Slide number appears in the footer right on every slide including title and closing slides.
RULE
Confidentiality label ("Confidential" or "Internal") appears in the footer left on every slide. Precedes the source if both are present, separated by " · ".
RULE
Never use more than 3 levels of hierarchy on a single slide. If depth is needed, split into multiple slides.
RULE
Bullet points are sentence fragments, not full sentences. No ending punctuation. Max 8 words per bullet.
RULE
Maximum 5 bullet points per column. If more are needed, consider a table or a different layout.
RULE
Charts and diagrams always have a title (the insight, not the chart type) and a source. Axis labels use the caption type (8pt).
Color usage rules
RULE
Color encodes meaning, not decoration. Never use --success, --warning, --danger, or --info for aesthetic reasons alone.
RULE
--success (#3a6b3a) = healthy, on target, resolved, passing. Use for SLO met, incident resolved, test green.
RULE
--warning (#b8882a) = degraded, at risk, approaching limit. Use for error budget < 50%, elevated latency, soft threshold breach.
RULE
--danger (#8b2e2e) = incident, outage, failed, critical. Use for P0/P1 active, SLO breached, hard threshold exceeded.
RULE
--info (#2e5c8b) = neutral informational, in progress, planned. Use for upcoming changes, non-urgent notices.
RULE
--accent (#b8882a) is reserved for structural decoration only: section divider rule, active nav marker. Never for data encoding.
RULE
On dark backgrounds (title slide, closing slide), body text uses --bg (#f7f3eb), secondary text uses --muted (#9a7f5e).
Header & footer spec
ZoneLeft contentRight contentBackground
Header band Brand name — uppercase, 7pt, 500 weight, ls 0.10em Deck name · Month Year — 7pt, muted --fg (#2c1f0e) on content slides. Transparent on title/closing.
Footer band Confidential · Source: [origin] — 7pt, muted Slide number — 7pt, muted Transparent always. Separated from content by 1pt --border rule.
03 — Slides

Layout catalogue

Eight named layouts defined in the Google Slides master. Each layout has a fixed name used by the Apps Script to reference and instantiate it. Layouts are described here — not mocked up — because the actual implementation lives in the master file.

LAYOUT_TITLE title slide
Use once — first slide of every deck
Content fields
· deck_title — main headline, 22pt, --bg, max 2 lines
· deck_subtitle — context line, 11pt, --muted
· author — name, 7pt, --muted, uppercase
· date — Month Year, 7pt, --muted
Visual rules
· Full slide background: --fg (#2c1f0e)
· No header band visible
· Content centered vertically at 40% from top
· Footer band present (page number + confidential)
LAYOUT_EXEC_SUMMARY executive summary
Use once — second slide of every deck
Content fields
· insight_title — governing thought, 22pt
· bullets[] — 3 to 5 items max, each with optional semantic color dot (success / warning / danger / none)
Visual rules
· Section label "Executive summary" above bullets, 8pt, --muted, uppercase, 1pt --border rule below
· Each bullet preceded by a 5pt circle in the semantic color (--fg if none specified)
· Single column, full content width
LAYOUT_SECTION section divider
Use to open each major section of the deck
Content fields
· section_index — e.g. "02 / 04", 8pt, --muted
· section_title — 22pt, --fg, max 2 lines
Visual rules
· Background: --bg-subtle (#f0ead9)
· 3pt vertical rule in --accent on the left edge of the content area
· No header band. Footer band present.
· Content vertically centered
LAYOUT_TWO_COL two column
Comparison, before/after, problem/solution
Content fields
· insight_title — governing thought, 22pt
· col_left.label — column header, 13pt, --muted, uppercase
· col_left.bullets[] — max 5 items, optional semantic color
· col_right.label + col_right.bullets[] — same structure
Visual rules
· Columns equal width, separated by 0.5cm gutter with a 1pt --border vertical rule
· Column labels sit directly above their content with a 1pt --border rule below
· Bullets use a 2pt left-side rule in the semantic color (--border if none)
LAYOUT_STAT stat callout
When the key message is a number. Use for KPIs, SLOs, metrics.
Content fields
· insight_title — governing thought, 22pt
· stats[] — 2 to 4 items, each with: value, unit, label, delta (optional), semantic color (optional)
Visual rules
· Each stat in a card: --bg-subtle fill, 1pt --border stroke, 0.3cm radius
· Value at 36pt in semantic color (--fg if none). Unit at 18pt, --muted.
· Label at 8pt, --muted, uppercase below value
· Delta (↑ / ↓ + text) at 8pt in --success or --danger below label
· Cards in equal-width row, 0.5cm gutter
LAYOUT_TIMELINE timeline
Incident timelines, roadmaps, process sequences
Content fields
· insight_title — governing thought, 22pt
· events[] — max 6 items, each with: timestamp, description, semantic color
Visual rules
· Two-column grid: timestamp (right-aligned, --muted, 8pt) + description (11pt, --fg)
· Description preceded by a 2pt left rule in the semantic color of the event
· Events separated by 0.4cm vertical gap
· No horizontal divider between events
LAYOUT_RECOMMENDATION recommendations
Action items, next steps, proposals with owners and dates
Content fields
· insight_title — governing thought, 22pt
· actions[] — max 5 items, each with: index, description, owner, due, semantic color
Visual rules
· Each action in a row: index (semantic color, 11pt, 500) + description (11pt, --fg) + owner · due (8pt, --muted, right-aligned)
· Row background: semantic color at 8% opacity, 1pt semantic border, 0.2cm radius
· Rows separated by 0.3cm gap
LAYOUT_CLOSING closing
Use once — last slide of every deck
Content fields
· closing_line — e.g. "Questions & next steps", 22pt, --bg
· contact — email or handle, 11pt, --muted
· deck_name — repeated from title slide, 7pt, --muted, bottom
Visual rules
· Full slide background: --fg (#2c1f0e), mirrors title slide
· 2cm × 2pt horizontal rule in --accent between closing_line and contact
· Content positioned at 45% from top, left-aligned
· deck_name at bottom left, 1.2cm from bottom edge
